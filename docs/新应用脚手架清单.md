# 新应用脚手架清单（可复用模板）

本文是**逐项勾选**的通用模板。每次上新应用时，按清单替换变量并打勾。

需要更深入的字段原理或完整配置项说明，见 `docs/宝塔apphub完整配置参考.md`。
真实项目的完整走查示例，见：
- `docs/substore-apphub-完整上架说明.md`（简单场景）
- `docs/sillytavern-apphub-完整上架说明.md`（含认证/白名单/安全默认值）

## 1. 目录脚手架

目标目录（将 `<appname>` 替换为应用名）：

```text
apphub/
  <appname>/
    app.json
    icon.png
    latest/
      docker-compose.yml
      .env
```

命令：

```powershell
New-Item -ItemType Directory -Force apphub/<appname>/latest
```

## 2. app.json 脚手架（字段清单）

必填最小字段：
1. `appid: -1`
2. `appname: <appname>`
3. `apptitle: <展示名>`
4. `apptype` / `appTypeCN`
5. `appversion`（至少 latest）
6. `appdesc`
7. `appstatus: 1`
8. `home` / `help`
9. `updateat: <unix_timestamp>`
10. `field`（至少包含 domain、allow_access、cpus、memory_limit + 业务字段）
11. `env`（必须包含 app_path、host_ip、cpus、memory_limit + 业务字段）
12. `volumes`

建议 `apptype` 对照：
1. BuildWebsite / 建站
2. Database / 数据库
3. Storage / 存储/网盘
4. Tools / 实用工具
5. Middleware / 中间件
6. AI / AI/大模型
7. Media / 多媒体
8. Email / 邮件/邮局
9. DevOps / DevOps
10. System / 系统

### 2.1 必填 4 项 field 的**官方话术**（直接抄）

**不要自己改写文案**，这些是面板前端拼接展示时依赖的约定：

```json
{
    "attr": "domain",
    "name": "域名",
    "type": "textarea",
    "default": "",
    "suffix": "浏览器访问的域名,非必填",
    "unit": ""
},
{
    "attr": "allow_access",
    "name": "允许外部访问",
    "type": "checkbox",
    "default": true,
    "suffix": "允许直接通过主机IP+端口访问，如果您设置了域名请不要勾选这里",
    "unit": ""
},
{
    "attr": "cpus",
    "name": "cpu核心数限制",
    "type": "number",
    "default": 0,
    "suffix": "0为不限制,最大可用核心数为: ",
    "unit": ""
},
{
    "attr": "memory_limit",
    "name": "内存限制",
    "type": "number",
    "default": 0,
    "suffix": "0为不限制,最大可用内存为: ",
    "unit": ""
}
```

**重点**：`cpus` / `memory_limit` 的 `suffix` 末尾的**冒号+空格**是面板拼接主机实际容量的锚点，不要漏，不要改成中文冒号。

### 2.2 校验命令

```powershell
Get-Content "apphub/<appname>/app.json" | ConvertFrom-Json | Out-Null
Write-Output "app.json OK"
```

## 3. docker-compose.yml 脚手架

最小模板（按应用改 image/ports/volumes/env）：

```yaml
services:
  <appname>:
    image: <docker_image>:<tag>        # 显式 tag，不要裸 image
    restart: always
    deploy:
      resources:
        limits:
          cpus: ${CPUS}
          memory: ${MEMORY_LIMIT}
    environment:
      - <APP_ENV_1>=${APP_ENV_1}
      - <APP_ENV_2>=${APP_ENV_2}
    ports:
      - ${HOST_IP}:${APP_PORT}:<container_port>
    volumes:
      - ${APP_PATH}/data:<container_data_path>
    labels:
      createdBy: "bt_apps"
    networks:
      - baota_net

networks:
  baota_net:
    external: true
```

必须保留：
1. `labels.createdBy: "bt_apps"`
2. `cpus/memory` 限制字段
3. `${HOST_IP}` 端口前缀
4. `${APP_PATH}` 数据目录前缀
5. **image 显式 tag**（即使用 `:latest` 也要写出来）

禁止/不推荐：
1. `container_name` —— 会导致宝塔状态识别异常
2. `build: .` —— 宝塔走镜像拉取，不走本地 build
3. 命名卷（如 `my_vol:/data`）—— 用 `${APP_PATH}/xxx` 绑定挂载

## 4. .env 脚手架

规则：
1. 变量名全部大写。
2. 值全部留空。
3. 与 `app.json` 的 `env.key`（转大写）严格一致。
4. 必含：`HOST_IP`、`CPUS`、`MEMORY_LIMIT`、`APP_PATH`。

示例：

```env
APP_PORT=
APP_ENV_1=
APP_ENV_2=
HOST_IP=
CPUS=
MEMORY_LIMIT=
APP_PATH=
```

## 5. icon.png 脚手架

要求：
1. PNG。
2. 推荐 100x100。
3. 文件名固定 `icon.png`。

从上游仓库扒图标的通用思路：
1. 先找 `public/img/logo.png`、`assets/logo.png`、`docs/images/logo.png`
2. 其次 `apple-touch-icon-*.png`（通常正方形，高清）
3. 最后 favicon.ico（需要转 PNG）

快速缩放到 100x100（PowerShell）：

```powershell
Add-Type -AssemblyName System.Drawing
$src = [System.Drawing.Image]::FromFile("apphub/<appname>/icon.png")
$dst = New-Object System.Drawing.Bitmap 100, 100
$g = [System.Drawing.Graphics]::FromImage($dst)
$g.InterpolationMode = [System.Drawing.Drawing2D.InterpolationMode]::HighQualityBicubic
$g.SmoothingMode = [System.Drawing.Drawing2D.SmoothingMode]::HighQuality
$g.PixelOffsetMode = [System.Drawing.Drawing2D.PixelOffsetMode]::HighQuality
$g.Clear([System.Drawing.Color]::Transparent)
$g.DrawImage($src, 0, 0, 100, 100)
$dst.Save("apphub/<appname>/icon.png", [System.Drawing.Imaging.ImageFormat]::Png)
$g.Dispose(); $src.Dispose(); $dst.Dispose()
```

## 6. 上线前检查清单（逐项勾选）

### 6.1 结构与规范

1. [ ] 目录结构正确。
2. [ ] app.json 可解析（PowerShell 校验命令通过）。
3. [ ] latest 下 `docker-compose.yml` 与 `.env` 同时存在。
4. [ ] `.env` 与 `app.json env` 键名一致（大写对应）。
5. [ ] `appname` 和目录名、compose service 名三者一致。
6. [ ] 图标存在、是 PNG、推荐 100x100。
7. [ ] 只保留需要导入的应用目录（删掉示例 alist/deeplx）。

### 6.2 compose 规范

8. [ ] `labels.createdBy: "bt_apps"` 存在。
9. [ ] `image` 有显式 tag（`:latest` 或具体版本）。
10. [ ] 端口用 `${HOST_IP}:${PORT}:容器端口` 格式。
11. [ ] 卷挂载用 `${APP_PATH}/xxx` 前缀。
12. [ ] 没用 `container_name`、`build`、`hostname`。
13. [ ] `deploy.resources.limits.cpus/memory` 用 `${CPUS}` / `${MEMORY_LIMIT}`。
14. [ ] 接入 `baota_net` 外部网络。

### 6.3 字段文案对齐官方（重要）

15. [ ] `domain.suffix` = `"浏览器访问的域名,非必填"`（半角逗号）
16. [ ] `allow_access.suffix` 包含"如果您设置了域名请不要勾选这里"
17. [ ] `cpus.suffix` = `"0为不限制,最大可用核心数为: "`（末尾冒号+空格）
18. [ ] `memory_limit.suffix` = `"0为不限制,最大可用内存为: "`（末尾冒号+空格）

### 6.4 应用默认安全机制（重要，见 §9）

19. [ ] 调研过应用有没有默认白名单/ACL/认证？
20. [ ] 如果有，已经在 field 里暴露为可配置字段？
21. [ ] 默认密码类字段的 `default` 值足够醒目（如 `please_change_me`）？

### 6.5 元数据

22. [ ] `updateat` 已更新为当前 Unix 时间戳。
23. [ ] `appdesc` 1~2 句话能让人看懂这是什么。

## 7. Git 提交流程脚手架

```powershell
git status --short

# 首次可设置仓库级身份
git config user.name "<github_name>"
git config user.email "<github_noreply_email>"

git add apphub/<appname>
git commit -m "feat: add <appname> app"

git remote set-url origin https://github.com/<user>/apphub.git
git push -u origin main
```

**commit 信息约定**：
- 新增应用：`feat: add <appname> app`
- 修复：`fix(<appname>): <issue>`
- 对齐风格：`refactor: polish <appname> app.json to align with official style`
- 文档：`docs: <...>`
- 功能增强：`feat(<appname>): <capability>`

## 8. 宝塔导入与更新脚手架

导入：
1. Docker 应用商店 → 外部仓库 → 导入你的仓库地址。
2. 同步/刷新。
3. 安装应用并填写参数。
4. **注意**：安装表单**顶部**有一个"名称"输入框，默认值是 `appname_<随机后缀>`，想改展示名就在这里改（app.json 不支持自定义这个默认值）。

后续更新：
1. 改模板文件。
2. 更新 `updateat`。
3. 提交推送。
4. 面板同步外部仓库。
5. 需要镜像更新时，执行"更新镜像/重建容器"。

**关键**：**模板更新不会自动同步到已安装的实例**。已装实例要吃新模板：
- 方案 A：热补丁（手动改已部署的 compose/.env，保留数据）
- 方案 B：备份数据 → 卸载重装 → 恢复数据

## 9. 应用默认安全机制处理要点（重要经验）

上线前**一定要**跑一次完整访问流程，而不是只看容器启动成功。很多应用有自己的默认安全机制，第一次通过域名访问会直接被拦住。

### 9.1 常见拦截类型

| 应用类型 | 默认拦截 | 典型现象 |
|---|---|---|
| LLM/Web UI | IP 白名单 | `Forbidden`、`blocked` |
| 数据库管理器 | 默认密码/Token | 需要 setup 页面 |
| API 网关 | 默认不启用认证 | 公网可写接口 |
| 看板/监控 | 默认弱密码 | `admin/admin` |

### 9.2 处理原则

1. **优先在 field 里暴露可配置开关**，而不是写在文档里让用户装完手动改 `config.yaml`。
2. **默认配置应该是"域名访问+带认证"能直接用**，而不是"只有 127.0.0.1 能访问"。
3. **密码/Token 类字段**要给醒目的 default（如 `please_change_me`），在 suffix 里提醒必改。
4. **可以叠加的认证方式**（如 HTTP 基础认证 + 应用自身账号系统）都暴露为开关，让用户按场景选。

### 9.3 案例参考

`apphub/sillytavern/` 的 `app.json` 暴露了 5 个安全相关字段：
- `whitelist_mode`（默认关，走域名必须）
- `basicauth_mode` + 用户名 + 密码（默认开）
- `enable_user_accounts`（可选，SillyTavern 账号系统）

## 10. 常见报错速查

1. 报 `docker-compose.yml 不存在`：通常是 `.env` 缺失或键名不匹配。
2. 安装后前端可开、后端连不上：检查 API 路径或应用是否需要首次 `?api=` 引导。
3. 面板看不到更新：`updateat` 未递增或仓库未同步。
4. 容器不受面板托管：compose 未加 `createdBy: "bt_apps"`。
5. 访问返回 `Forbidden` / `blocked`：应用默认白名单/ACL 没处理，补 env 关掉或走反代（见 §9）。
6. 展示容量后缀显示异常（如 `0为不限制8`）：`cpus/memory_limit.suffix` 末尾漏了冒号+空格。
7. 安装时端口报占用：`env.key` 的 `type` 是否设为 `port`？`port` 类型才会做端口检测。
8. 已装实例名叫 `appname_xxx` 改不了：这是面板硬编码，安装时在表单顶部"名称"框修改。
